# Production Regressions Design

## Scope

Fix four regressions without changing existing accounts, passwords, business
records, uploaded files, or the public Lucky reverse-proxy address:

1. Engineering archive preview and download fail.
2. User avatars and the application logo do not display.
3. Payment dashboard cards have no visible hover color feedback.
4. System settings has no entry for the shared official DeepSeek API key.

## Uploaded File Compatibility

The upload directory still contains the historical files. The regression comes
from the hardened download route accepting only new UUID filenames, while
existing database records use the former `timestamp-random.extension` format.

The server will accept both formats through strict allowlists:

- New UUID v4 names generated by the current upload endpoint.
- Historical names matching a 13-digit timestamp, a numeric random suffix, and
  an already-approved extension.

The route will continue rejecting path separators, traversal attempts,
unsupported extensions, and arbitrary filenames. It will serve files inline by
default so authenticated images and PDFs can render in the application.
Explicit downloads will use `?download=1` and retain attachment disposition.
New uploads will continue using UUID filenames; no database rewrite or file
rename is required.

## Payment Dashboard Feedback

The existing hover classes are compiled but nearly invisible because their
background opacity is only 10 to 20 percent. Each summary card will use a
static, visibly tinted hover background and border matching the card's semantic
color. Existing scale, shadow, and dark-mode behavior will remain.

## DeepSeek Configuration

The DeepSeek key is a global system secret. It must not be stored in the
existing public settings JSON because every authenticated user can retrieve
that record.

An additive migration will create a dedicated `system_secrets` table. The key
will be encrypted with AES-256-GCM using a key derived from the existing stable
`SESSION_SECRET` plus a fixed application context. The plaintext key will
exist only while processing an administrator request or calling DeepSeek.

Admin-only endpoints will:

- Return configured/unconfigured status and a masked suffix, never plaintext.
- Save or replace a validated key.
- Explicitly clear the saved key.

The system settings dialog will add an AI configuration tab. A blank field will
leave the current key unchanged. The official DeepSeek gateway will resolve the
database key for each request and fall back to `DEEPSEEK_API_KEY` for deployment
compatibility. Existing runtime model registration remains unchanged, so newly
released official DeepSeek model IDs can still be added without a code release.

## Failure Handling And Rollback

- Missing or invalid historical files return the existing safe 404 response.
- Invalid secret input returns 400; non-admin access returns 403.
- Encryption or database failures return a generic server error and never log
  or return the key.
- The schema change is additive, so the current production version can run
  against the upgraded database if rollback is needed.
- Deployment will take a final database and uploads backup, retain the current
  image/version, verify local and public health, and restore the prior version
  if any required check fails.

## Verification

Automated tests will cover legacy and UUID file access, inline preview,
attachment download, filename rejection, admin authorization, secret
non-disclosure, encryption round trips, runtime key resolution, settings UI
source wiring, and visible hover classes. The release must also pass the full
test suite, TypeScript/Vite production build, and release verification script
before Synology deployment.
